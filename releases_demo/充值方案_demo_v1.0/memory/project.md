# 项目记忆

## 项目定位

- 智能外呼中台纯前端原型，使用 HTML、CSS 和原生 JavaScript。
- 不需要构建或安装依赖，入口为根目录 `index.html`。
- 页面通过 `window.Pages[pageKey]` 注册，由 `js/nav.js` 负责切换。
- 默认打开系统管理下的租户管理，page key 为 `sys-tenant`。

## 工作原则

- 先理解现有代码和业务逻辑，再修改。
- 非必要不修改现有板块内容，不做无关重构。
- 业务功能、Mock、说明文档、长期记忆和标注工具分别维护。
- 页面标注规则只维护在 `.clauderules`，不在其他文档重复定义。

## 角色权限背景

以下内容来自早期原型设计，用于理解页面角色，不代表生产权限已经最终确认：

| 角色 | 数据范围 | 操作范围 |
| --- | --- | --- |
| 超级管理员 | 所有租户数据 | 早期规则为只能操作自己创建的任务 |
| 租户管理员 | 本租户数据 | 可查询和操作本租户创建的任务及数据 |
| 租户运营 | 本租户数据 | 仅查看，不允许业务操作 |

当前原型右上角展示“产品-超管”。除产品超管外的按钮显隐、数据权限及操作权限仍以 `memory/open-items.md` 中的待确认项为准。

## 运行依赖

`index.html` 的脚本顺序为 Mock 数据、公共逻辑、页面模块、应用工具、导航、标注数据和标注运行时。`tools/fix_report_clue.py` 是历史维护脚本，当前引用的 `js/pages/report-clue.js` 不在本项目中，不属于运行依赖。

## 当前计费实现

- 外部充值只提供金额，进入租户统一资金账户，不区分大模型、小模型或供应商。
- 当前价格规则维护在 `MockTenantPriceRules`，每个租户只有大模型、小模型两条 `MODEL_DEFAULT` 默认价格。
- `providerCode` 在模型默认价规则中为空；供应商目录 `MockBillingVendors` 仅为未来扩展保留。
- 未来可增加 `PROVIDER_OVERRIDE` 规则，价格解析优先使用供应商覆盖价，没有覆盖时回退模型默认价。
- 租户主数据不直接保存单价，避免存在两套价格来源。
- 冻结任务维护模型类型、供应商、冻结分钟数和单价快照；已冻结任务不受后续调价影响。
- 冻结任务同时维护任务状态、冻结状态、释放时间和释放原因。
- 任务完成、终止时立即释放冻结；冻结中超过 24 小时自动释放，且释放操作必须幂等。
- 资金余额 = 已生效通话充值总额 - 手工扣减金额 - 已消费金额。
- 冻结总额 = 所有有效冻结任务的冻结分钟数 × 各任务单价快照。
- 统一可用金额 = max（资金余额 - 冻结总额，0）。
- 某模型的通话余额分钟数 = 资金余额 ÷ 当前模型默认价。
- 某模型的可用分钟数 = 统一可用金额 ÷ 当前模型默认价。
- 大模型和小模型分钟数都是同一金额池的换算视图，禁止相加。
- 租户列表通过“计费配置”按钮打开独立配置弹窗，只修改大模型、小模型默认价及状态。
- 完整价格、冻结和分钟数据在充值管理抽屉的只读计费明细中展示。
- 充值管理抽屉使用”计费明细”、”充值单管理”和”余额调整”三个 Tab；租户信息位于 Tab 上方并由三页共用。
- 充值单管理 Tab 负责充值单关联、预览和历史记录，计费明细 Tab 只读展示模型计费结果，余额调整 Tab 独立管理手工扣减流水。
- 充值单采用逐单生效：关联后为待生效，历史记录中的对应”生效”按钮是唯一生效入口。
- 计费汇总只读取已生效充值单；待生效充值单既不延长有效期，也不增加通话余额。
- 手工扣减通过余额调整 Tab 的”手工调整”按钮发起弹窗，只校验资金余额，不关联充值单、不占用冻结金额。
- 扣减后若资金余额小于冻结总额，已有冻结保持不变，可用金额按 0 计算，不再设置独立的冻结覆盖不足全局状态。
- 手动导入按号码数、单号码预计分钟和租户模型供应商有效单价计算本次冻结；不超过可用金额时允许导入。
- 超限时拒绝导入，并展示按当前计费上下文计算出的最大可导入号码数。
- 历史关联充值单操作列仅提供”生效”按钮，已生效单据显示”已生效”状态，不提供退款扣减入口。
- 可用余额仅保留为隐藏字段，页面对外展示大模型、小模型分钟数。
- 总部类型租户不设特殊代码分支，通过配置超大额充值单（¥999,999,999，有效期 2026-01-01 ~ 2099-12-31）实现几乎无限的可用分钟数。
